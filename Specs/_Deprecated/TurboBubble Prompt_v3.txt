# TurboBubble - Revit 2025 C# Addin Prompt

Create a Revit 2025 C# addin with the following specifications:

## Basic Requirements
- Target framework: .NET 8.0
- No namespace (flat class structure)
- Public class name: `TurboBubble`
- Implements `IExternalCommand`
- Only display dialog boxes when the script fails (no success messages)
- Use extracted helper methods for better organization and maintainability
- Define constant: `private const double INCHES_TO_FEET = 1.0 / 12.0;`

## View Restrictions
- Only works in Floor Plan or Reflected Ceiling Plan (RCP) views
- Show error dialog and cancel if run in any other view type

## Code Organization

The Execute method should be broken down into focused helper methods:

### Main Execute Method Flow
1. Validate prerequisites (tag type exists, correct view type)
2. Get user inputs (select tag, get fixture, pick flip point)
3. Calculate geometry (transforms, condition, flip state, offsets)
4. Create elements (wire and lighting fixture tag)

The method should be streamlined to show clear logical sections rather than numbered steps.

## User Interaction Flow
1. Prompt user to select a Lighting Fixture Tag that tags a lighting fixture
2. Automatically determine the Lighting Fixture from the tag's host element
3. Prompt user to click anywhere in the view to establish flip direction
4. Create an MEP wire with specific geometry (potentially flipped based on click position)
5. Create a second Lighting Fixture Tag (AL_Tag_Lighting Fixture (Switchleg)) hosted to the fixture (potentially flipped based on click position)
5. Prompt user if they want to mirror the wire and tag across the fixture's axis

## Local Coordinate System
All elements are created in a local coordinate system defined by the Lighting Fixture's orientation:

### Getting the Fixture from the Tag
After the user selects a tag:
```csharp
ICollection<Element> taggedElements = fixtureTag.GetTaggedLocalElements();
FamilyInstance fixture = taggedElements.FirstOrDefault() as FamilyInstance;
```
Validate that the fixture exists and is in the OST_LightingFixtures category.

### Establishing the Coordinate System
- Use `fixture.GetTransform().BasisX` to get the fixture's actual X-axis direction
- Calculate rotation angle: `Math.Atan2(basisX.Y, basisX.X)` - this gives the rotation relative to project north
- Create transforms:
  - `localToGlobal`: `Transform.CreateRotationAtPoint(XYZ.BasisZ, rotation, fixturePoint)`
  - `globalToLocal`: `Transform.CreateRotationAtPoint(XYZ.BasisZ, -rotation, fixturePoint)`

## Determine Condition Based on Tag Position
Transform both fixture and tag positions to local coordinates, then determine which condition applies:

1. Calculate difference: `diff = tagLocalPosition - fixtureLocalPosition`
2. Compare absolute values: `absDiffX` vs `absDiffY`
3. Determine condition:
   - **Condition 1 (Right)**: `absDiffX > absDiffY` AND `diff.X > 0`
   - **Condition 2 (Left)**: `absDiffX > absDiffY` AND `diff.X < 0`
   - **Condition 3 (Up)**: `absDiffY > absDiffX` AND `diff.Y > 0`
   - **Condition 4 (Down)**: `absDiffY > absDiffX` AND `diff.Y < 0`

## Flip State Determination
After determining the condition, prompt the user to click anywhere in the view. The position of this click relative to the fixture tag determines whether the wire geometry should be flipped.

### Setting Up Point Selection
To allow point picking without requiring a pre-existing work plane:
```csharp
// Save original sketch plane (may be null)
SketchPlane originalSketchPlane = activeView.SketchPlane;

// Create temporary sketch plane at fixture elevation
using (Transaction trans = new Transaction(doc, "Set Temporary Sketch Plane"))
{
    trans.Start();
    Plane sketchPlane = Plane.CreateByNormalAndOrigin(XYZ.BasisZ, new XYZ(0, 0, fixturePoint.Z));
    SketchPlane tempSketchPlane = SketchPlane.Create(doc, sketchPlane);
    activeView.SketchPlane = tempSketchPlane;
    trans.Commit();
}

// Pick point
XYZ secondPoint = uidoc.Selection.PickPoint("Click anywhere to establish flip direction");

// Clean up
using (Transaction trans = new Transaction(doc, "Restore Sketch Plane"))
{
    trans.Start();
    if (originalSketchPlane != null)
    {
        activeView.SketchPlane = originalSketchPlane;
    }
    doc.Delete(tempSketchPlane.Id);
    trans.Commit();
}
```

### Flip Logic
Transform the second point to local coordinates and compare to the tag position:
```csharp
XYZ secondPointLocal = globalToLocal.OfPoint(secondPoint);
XYZ tagDiff = secondPointLocal - tagLocalPosition;

bool flipState = false;
if (condition == 1 || condition == 2)
{
    // For left/right conditions, check Y position of second point relative to tag
    flipState = tagDiff.Y < 0; // Flip if second point is below the tag
}
else // condition == 3 || condition == 4
{
    // For up/down conditions, check X position of second point relative to tag
    flipState = tagDiff.X < 0; // Flip if second point is to the left of the tag
}
```

### Applying Flip State
After calculating all offsets, apply flip conditionally:
```csharp
if (flipState)
{
    if (condition == 1 || condition == 2)
    {
        // Conditions 1 & 2 (Right/Left): Only negate Y values
        vertex2YOffset = -vertex2YOffset;
        vertex3YOffset = -vertex3YOffset;
        tagYOffset = -tagYOffset;
    }
    else // condition == 3 || condition == 4
    {
        // Conditions 3 & 4 (Up/Down): Only negate X values
        vertex2XOffset = -vertex2XOffset;
        vertex3XOffset = -vertex3XOffset;
        tagXOffset = -tagXOffset;
    }
}
```

## Wire Geometry Requirements
The wire must have three vertices, all in the X/Y plane (no Z-direction changes):

### Vertex 1
- Connected directly to the Lighting Fixture's electrical connector (not a discrete point)
- Find connector using: `fixture.MEPModel.ConnectorManager.Connectors` where `connector.Domain == Domain.DomainElectrical`

### Vertex 2 (Local Offsets from Fixture)
- **Condition 1 (Right)**: X = -0.68", Y = +5.19"
- **Condition 2 (Left)**: X = +0.68", Y = +5.19"
- **Condition 3 (Up)**: X = +7.34375", Y = -0.71875"
- **Condition 4 (Down)**: X = +7.34375", Y = +0.71875"

### Vertex 3 (Local Offsets from Fixture)
- **Condition 1 (Right)**: X = +2.5", Y = +6.5"
- **Condition 2 (Left)**: X = -2.5", Y = +6.5"
- **Condition 3 (Up)**: X = +9.21875", Y = +5.0625"
- **Condition 4 (Down)**: X = +9.21875", Y = -5.0625"

### Coordinate Transformation for Vertices
1. Define offsets in local space: `XYZ vertexLocal = new XYZ(xOffset, yOffset, 0)`
2. Convert inches to feet: divide by 12.0
3. Transform to global: `XYZ vertex = fixturePoint + localToGlobal.OfVector(vertexLocal)`

## Wire Creation

- Parameters:
  - `wireTypeId`: First available WireType in document
  - `vertices`: `IList<XYZ>` containing vertex2 and vertex3 only
  - `startConnector`: The electrical connector from the fixture
  - `endConnector`: null

## Lighting Fixture Tag Requirements
- **IMPORTANT**: Check if tag type "AL_Tag_Lighting Fixture (Switchleg)" exists at the very beginning of Execute
- If tag type is not found, show error and return Result.Cancelled immediately (before any user interaction)
- Tag is hosted to the Lighting Fixture (not the wire)
- Tag position based on the condition:

### Lighting Fixture Tag Position (Local Offsets from Fixture)
- **Condition 1 (Right)**: X = +8.8125", Y = +7.5"
- **Condition 2 (Left)**: X = -8.8125", Y = +7.5"
- **Condition 3 (Up)**: X = +12.625", Y = align with fixture tag's Y in local coordinates
- **Condition 4 (Down)**: X = +12.625", Y = align with fixture tag's Y in local coordinates

### Y-Alignment for Conditions 3 & 4
```csharp
double fixtureTagYLocal = tagLocalPosition.Y - fixtureLocalPosition.Y;
tagLocalOffset = new XYZ(tagXOffset, fixtureTagYLocal, 0);
```

### Lighting Fixture Tag Creation and Rotation
1. Tag type is already validated at the start of Execute and passed as parameter
2. Calculate position: `tagPosition = fixturePoint + localToGlobal.OfVector(tagLocalOffset)`
3. Create tag: `IndependentTag.Create(doc, viewId, fixtureRef, false, TagMode.TM_ADDBY_CATEGORY, TagOrientation.Horizontal, tagPosition)`
   - Note: Use `fixtureRef` (Reference to fixture) instead of wireRef
4. Set type: `fixtureTag.ChangeTypeId(tagTypeId)`
5. Set position explicitly: `fixtureTag.TagHeadPosition = tagPosition`
6. Rotate tag around its own position: `ElementTransformUtils.RotateElement(doc, fixtureTag.Id, rotationAxis, fixtureRotation)`
   - Rotation axis: `Line.CreateBound(tagPosition, tagPosition + XYZ.BasisZ)`

```csharp
private ElementId GetWireTypeId(Document doc)
{
    WireType wireType = new FilteredElementCollector(doc)
        .OfClass(typeof(WireType))
```

### GetLightingFixtureSwitchlegTagTypeId(Document doc)
```csharp
private ElementId GetLightingFixtureSwitchlegTagTypeId(Document doc)
{
    FamilySymbol tagType = new FilteredElementCollector(doc)
        .OfClass(typeof(FamilySymbol))
        .OfCategory(BuiltInCategory.OST_LightingFixtureTags)
        .Cast<FamilySymbol>()
        .FirstOrDefault(fs => fs.FamilyName == "AL_Tag_Lighting Fixture (Switchleg)");

    return tagType?.Id ?? ElementId.InvalidElementId;
}
```

## Critical Implementation Notes

1. **Rotation Handling**: The fixture's rotation must be obtained from its transform (`GetTransform().BasisX`), NOT from `LocationPoint.Rotation`, to correctly account for project north rotation.

2. **Coordinate Transformation**: All geometry is defined in a local coordinate system where:
   - Local X = fixture's rotated X-axis
   - Local Y = fixture's rotated Y-axis
   - Transform local offsets to global using: `fixturePoint + localToGlobal.OfVector(localOffset)`
   - Use `Transform.Inverse` instead of creating separate inverse transform

3. **Tag Rotation**: Tags must be explicitly rotated around their own position after placement to match the fixture's orientation.

4. **Transaction Scope**: Wire creation and tag creation should be in separate transactions to allow the wire to be created even if tagging fails.

5. **Method Organization**: Break down the Execute method into focused helper methods for better maintainability and testability. Group related operations logically (validation, user input, calculation, creation).

6. **Error Handling**: Use exception-based error handling with helper methods throwing InvalidOperationException and a centralized try-catch in Execute. Use throw expressions with null-coalescing operators for cleaner validation.

7. **Constants**: Use `INCHES_TO_FEET` constant instead of repeated division by 12.0.

8. **API Updates**: Use `Id.Value` (returns long) instead of deprecated `Id.IntegerValue` (returns int).

9. **Resource Cleanup**: Use try-finally blocks to ensure sketch plane cleanup even if user cancels point selection.

10. **LINQ Usage**: Simplify collection filtering with LINQ methods like `Any()` and `FirstOrDefault()` for cleaner code. Use expression-bodied members where appropriate.

11. **Null-Coalescing Throw**: Use `?? throw` pattern for cleaner null checks: `obj ?? throw new Exception()`

12. **Tuple Deconstruction**: Use tuple deconstruction for cleaner multi-value assignments in switch statements and flip logic.

13. **Ternary Operators**: Use nested ternary operators judiciously for simple conditional logic to reduce code verbosity.

14. **Parameter Reduction**: Pass higher-level objects (e.g., FamilyInstance) instead of derived values (e.g., XYZ fixturePoint) when the method needs to calculate multiple properties.

15. **Transform.Inverse**: Use `Transform.Inverse` property instead of creating a separate inverse transform with negative rotation.
